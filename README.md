## What this is

A shared Jenkins pipeline that automatically scans Android APK files using MobSF static analysis on every push. Designed specifically for organisations that receive APK files from external suppliers and need to audit them before internal distribution.

This is the third pipeline in the DevSecOps suite:

| Pipeline | Scans | Repository |
|---|---|---|
| [devsecops-pipeline](https://github.com/r0ms3c/devsecops-pipeline) | Application source code (PHP, .NET, Node.js) | Semgrep · Dependency-Check · SonarQube |
| [iac-security-pipeline](https://github.com/r0ms3c/iac-security-pipeline) | Infrastructure as Code (Terraform, Ansible, Docker, K8s) | Trivy · Checkov · ansible-lint · tflint · Hadolint |
| **mobile-security-pipeline** | Android APK files from suppliers | **MobSF** |

## What it finds

| Category | Examples |
|---|---|
| **Hardcoded secrets** | API keys · passwords · tokens embedded in Java/Kotlin code |
| **Dangerous permissions** | RECORD_AUDIO · READ_CONTACTS · READ_EXTERNAL_STORAGE · SEND_SMS |
| **Manifest issues** | debuggable=true · backup enabled · exported activities without permission |
| **Weak cryptography** | MD5/SHA1 · ECB mode · insufficient key sizes |
| **Network security** | HTTP cleartext traffic · certificate pinning disabled · trust-all certificates |
| **Third-party trackers** | Analytics, advertising, and telemetry SDKs embedded in the APK |
| **Certificate issues** | Signed with debug certificate · weak signature algorithm |
| **Code issues** | SQL injection · logging sensitive data · insecure random |

## Architecture

```
Supplier delivers APK
        │
        │ committed to GitLab repository
        ▼
GitLab webhook
        │
        ▼
Jenkins (shared pipeline)
        │
        ├── Checkout mobile repository
        ├── Detect APK file (auto-detect or APK_PATH parameter)
        │
        ├── Upload APK → MobSF REST API
        │         POST /api/v1/upload
        │         Returns: scan hash (MD5 of APK)
        │
        ├── Run static analysis
        │         POST /api/v1/scan
        │         MobSF decompiles APK, analyses manifest,
        │         code, permissions, certificate
        │
        ├── Download reports
        │         GET /api/v1/report_json  → mobsf-report.json
        │         GET /api/v1/download_pdf → mobsf-report.pdf
        │
        ├── Parse findings
        │         Score · HIGH findings · permissions
        │         Secrets · trackers · manifest issues
        │
        ├── Post GitLab commit comment
        │
        ├── Generate mobile-security-report.html
        │
        └── Security Gate
              ├── Score < 50         → FAIL
              ├── HIGH findings > 0  → FAIL
              └── Secrets > 0        → FAIL
```

## Pipeline stages

| Stage | Description |
|---|---|
| 1 · Validate | Check required parameters |
| 2 · Checkout | Copy report generator to /tmp, fetch mobile repository |
| 3 · Detect APK | Auto-detect *.apk or use APK_PATH parameter |
| 4 · Upload | Upload APK to MobSF REST API, receive scan hash |
| 5 · Scan | Trigger MobSF static analysis, capture security score |
| 6 · Download | Download JSON and PDF reports, archive in Jenkins |
| 7 · Parse | Extract score, findings, permissions, secrets, trackers |
| 8 · Report | Generate mobile-security-report.html |
| 9 · Security Gate | Pass/fail based on score, HIGH findings, secrets |

---

## Security Gate thresholds

| Condition | Result |
|---|---|
| Security score < 50 | FAIL — return to supplier |
| HIGH severity findings > 0 | FAIL — critical issues must be fixed |
| Hardcoded secrets > 0 | FAIL — revoke credentials immediately |
| Score ≥ 50, no HIGH, no secrets | PASS |

---

## MobSF security score

MobSF scores APKs from 0 (insecure) to 100 (secure):

| Score | Interpretation |
|---|---|
| 70–100 | Good — follows Android security best practices |
| 50–69 | Acceptable — minimum to pass; plan fixes for next version |
| 0–49 | Fail — significant security issues; return to supplier |

## Onboarding a new supplier APK

Three steps, approximately 10 minutes:

**1 — Create the Jenkins job**

New Item → Pipeline. Add four String Parameters:

| Parameter | Description | Example |
|---|---|---|
| `REPO_NAME` | Repository name, no spaces | `supplier-app` |
| `GITLAB_PROJECT_ID` | Numeric GitLab project ID | `74` |
| `GITLAB_REPO_PATH` | Path after the domain | `group/supplier-app` |
| `APK_PATH` | Path to APK file (leave empty to auto-detect) | `./app-release.apk` |

Pipeline section: point to this repository, branch `*/main`, script path `Jenkinsfile`.

**2 — Create the GitLab repository and commit the APK**

```bash
git init
git add app-release.apk
git commit -m "Add APK v1.0.0 for security review"
git remote add origin <gitlab>/group/supplier-app.git
git push -u origin main
```

**3 — Add the GitLab webhook**

Settings → Webhooks → Add webhook:
- URL: `http://<jenkins>:8080/project/<job-name>`
- Secret token: your shared webhook token
- Events: Push + Merge Request

## How MobSF handles re-scans

MobSF identifies APKs by their MD5 hash. Uploading the same APK file twice returns the existing scan result — MobSF does not re-analyse identical files. A new scan is created automatically only when the supplier delivers a new version of the APK (different file = different hash).

## MobSF installation

MobSF is deployed as a standalone service on a dedicated server. See [docs/mobsf-installation.md](docs/mobsf-installation.md) for the complete installation guide.

| Item | Detail |
|---|---|
| Install path | `/opt/mobsf` |
| Service | systemd — `mobsf.service` |
| Port | 8000 |
| Python env | Poetry virtual environment |
| API auth | `X-Mobsf-Api-Key` header — derived from SHA256 of secret file |
